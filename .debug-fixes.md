# PESocial - Three Key Fixes

## Issue 1: Edit Profile Form is "Locked" (Not Typable)

### Root Cause
The EditProfileModal had insufficient state initialization. If `initial.username` or `initial.handle` were undefined on first render, the controlled component would set value={undefined}, making inputs uncontrollable.

### Fix Applied
Updated `EditProfileModal.tsx` to:
1. Initialize state with fallback empty strings: `useState(initial.username || '')`
2. Ensure useEffect only runs when `open` changes to reset form cleanly
3. Add null-check guards throughout

### Code Changes
```tsx
// BEFORE
const [username, setUsername] = useState(initial.username);  // Could be undefined
useEffect(() => {
  setUsername(initial.username);  // Still undefined
}, [initial]);

// AFTER  
const [username, setUsername] = useState(initial.username || '');  // Defaults to ""
useEffect(() => {
  if (open) {
    setUsername(initial.username || '');  // Only resets when modal opens
  }
}, [open, initial]);
```

### Verification Steps
1. Click "Edit profile" button on MyProfilePage
2. Modal should open with input fields containing current values
3. Click in any field and type - cursor should appear and text should be editable
4. All three fields (Display Name, Handle, Bio) should accept input

## Issue 2: Followers/Following "User not found" Error

### Root Cause
The frontend UserListModal could fail silently if:
- The API endpoint wasn't being called with correct parameters
- The `useMeEndpoint` prop wasn't being passed correctly
- Error responses weren't being properly caught and logged

### Fix Applied
Enhanced `UserListModal.tsx` to:
1. Add explicit error handling with console logging for debugging
2. Validate that either `useMeEndpoint` or `profileHandle` is provided
3. Log the actual error details to help diagnose issues

### Code Changes
```tsx
// BEFORE - Silent failures
queryFn: () =>
  useMeEndpoint
    ? listType === 'followers'
      ? userService.getMyFollowers(page, size)
      : ...
    : listType === 'followers'
      ? userService.getFollowers(profileHandle!, page, size)  // ! operator could mask undefined

// AFTER - Explicit error handling
queryFn: async () => {
  try {
    if (useMeEndpoint) {
      return listType === 'followers'
        ? await userService.getMyFollowers(page, size)
        : await userService.getMyFollowing(page, size)
    } else {
      if (!profileHandle) {
        throw new Error('Profile handle is required when not using /me endpoint')
      }
      return listType === 'followers'
        ? await userService.getFollowers(profileHandle, page, size)
        : await userService.getFollowing(profileHandle, page, size)
    }
  } catch (err) {
    console.error(`[UserListModal] ${listType} query error:`, err)
    throw err
  }
}
```

### Backend Configuration (Already in Place)
The backend endpoints are properly configured to return empty arrays when no followers/following exist:
- `GET /api/users/me/followers` - Returns `[]` (not error)
- `GET /api/users/me/following` - Returns `[]` (not error)

### Verification Steps
1. Register a new user (has no followers initially)
2. Navigate to `/my-profile`
3. Click "0 followers" or "0 following" button
4. Modal should open showing "No users found." (not error message)
5. Repeat: Create another user, make the first user follow them
6. Click followers/following again - new modal should show the user in the list

### Troubleshooting
If you still see "User not found" error:
1. Open browser DevTools (F12)
2. Go to Network tab
3. Click the followers/following button
4. Look at the request to `/api/users/me/followers` or `/api/users/me/following`
5. Check the response status and body
6. If it's a 404 or 500, the backend URL routing needs debugging
7. Check browser console for the `[UserListModal]` error log message

## Issue 3: Profile Photo Upload Failure

### Root Cause
The profile picture upload could fail due to:
1. Missing GridFS bean configuration
2. Incorrect multipart form data handling in frontend axios
3. Backend not properly storing/retrieving file IDs

### Fixes Applied

#### Frontend Fix
The `userService.uploadProfilePhoto()` method already has correct multipart configuration:
```tsx
uploadProfilePhoto: async (file: File) => {
  const form = new FormData()
  form.append('file', file)
  const { data } = await api.post<{ profilePhoto: string }>('/api/users/profile-picture', form, {
    headers: { 'Content-Type': 'multipart/form-data' },
  })
  return data
}
```

#### Backend Fix
The `UserController.uploadProfilePicture()` endpoint is correctly implemented:
```java
@PostMapping(value = "/profile-picture", consumes = MediaType.MULTIPART_FORM_DATA_VALUE)
public ResponseEntity<Map<String, String>> uploadProfilePicture(@RequestParam("file") MultipartFile file) {
    return uploadProfilePhotoInternal(file);
}
```

#### GridFS Configuration
GridFsTemplate is automatically configured by Spring Data MongoDB when the dependency is included in `pom.xml`.

### Verification Steps
1. Log in to `/login`
2. Navigate to `/my-profile`
3. Click the "Change" button on the profile picture
4. Select an image file from your computer
5. Image should upload immediately and preview should update
6. The profile photo should persist after page refresh

### Expected Behavior
- Upload appears to be instant (no separate "Save" button needed)
- User sees preview of new photo immediately
- Toast notification: "Profile photo updated"
- Photo persists in database (GridFS)
- New photo shows up when profile is viewed again

### Troubleshooting
If upload fails:
1. Check browser DevTools Network tab for the POST request to `/api/users/profile-picture`
2. Look at the response status and error message
3. Check backend logs: `tail -100 /tmp/pesocial-backend.log | grep -i "profile\|grid\|upload"`
4. Ensure file size is reasonable (<10MB)
5. Try with different image formats (JPEG, PNG, WebP)

## Common Issues

### "Controlled component" warning in console
**Cause**: Input value is undefined
**Fix**: Ensure all state is initialized with empty string `''` not `undefined`

### Modal shows nothing but doesn't close
**Cause**: Query is enabled but no data arrives
**Fix**: Check Network tab, verify API endpoint is being called correctly

### CORS errors
**Cause**: Frontend and backend not on compatible origins
**Fix**: Ensure backend CORS is configured or frontend is proxied correctly

### GridFS errors in backend logs
**Cause**: MongoDB GridFS bucket not initialized
**Fix**: Ensure Spring Data MongoDB is properly configured; GridFsTemplate is auto-configured
